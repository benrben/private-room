# Image Intelligence — Full Implementation Plan
**Date:** 2026-07-16 (rev 2, same day) · **Status:** design complete, awaiting go decision · **Scope:** everything — quick wins, medium, and all three ambitious capabilities

> **Rev 2 — sidecar-only architecture verified.** After the design pass, we re-verified the working tree: the Rust→Python migration (sidecar/MIGRATION-SPEC.md) is further along than the design scouts assumed. Confirmed in code: `ollama.rs` is now a **pure sidecar proxy** (every model call goes through `sidecar_post` — no direct Ollama HTTP anywhere in Rust); the native `agent_loop`/`chat_stream_tools` are **deleted** (`stream_answer` routes external-CLI or sidecar only, "no native fallback"); UI-driving turns ride the sidecar via the MCP image-block bridge in `room_mcp.rs` (implemented + tested). Rust's remaining role matches the migration spec: DB/SQLCipher access, tool execution (`exec_tool` via the bridge), process lifecycle, extraction. Consequences are folded into **D3, D9, W1, W4** below (marked *rev 2*), and they make the plan simpler: one pixel-fence choke point instead of two, one image-fitting implementation instead of a cross-language pair. Line-number anchors in the raw workflow specs predate this verification — implementers must re-anchor (e.g. `exec_tool` is now agent.rs:1211, `perceive_image` :2046, the post-answer grounding block :265–306; references to `agent_loop`/`chat_core` are to deleted code).

## How this plan was built

Three stages, all grounded in the current working tree and in the adversarially fact-checked research report (2026-07-16):

1. **Scouts** mapped the four subsystems every feature must build on: the Python sidecar (the app's sole AI service; PyInstaller onedir, 55 MB, no torch/onnx today), the DB schema + dual-registration migration rule, the model-lifecycle machinery (pull-progress UI, `vision_keep_alive`, one-big-model rule, `Lane::LocalLlm`), and the viewer/effects overlay plumbing.
2. **Eight designers** produced function-level specs, each verifying its external facts on the web (licenses, GGUF availability, ONNX exports, Ollama internals).
3. **Three adversarial critics** (16 GB reality + over-engineering; end-to-end completeness; privacy/licensing/supply-chain) attacked the merged plan. Their verdict: individually excellent, structurally unreconciled — with two privacy-gate violations, one production data-loss bug, and one layer of duplicated infrastructure. **All of their fixes are folded into this document.** Raw specs and critiques are preserved in the workflow transcripts for reference.

**Total effort: ~54–58 engineer-days** after reconciliation and the rev-2 sidecar consolidation (vs ~75 as independently specced), across seven shippable phases. Every phase is independently valuable, feature-gated, and degrades to today's behavior when its model is absent.

## Part 1 — Binding foundation decisions (the reconciliation)

These twelve decisions resolve every collision the critics found. They are **prerequisites**: no feature PR lands until the ones it touches are in place. The reconciliation itself (D1, D2, D5-partial, D8, D10) is ~3–4 engineer-days and is Phase 0 work.

**D1. `extracted_text` becomes sectioned — one shared helper.** Five workstreams write derived text into `files.extracted_text`, and two of the specced write modes destroy each other's output (Surya's whole-string replace erases captions/deep-scans; the caption truncate-at-marker erases anything after it). Fix: one ~40-line `upsert_text_section(existing, begin_marker, end_marker, body)` in `db/files.rs` (modeled on DC2's correctly-scoped design) plus a fixed marker registry:
- `(text recognized from scan)` — OCR section (Apple and Surya both use this exact prefix; Surya replaces only this section)
- `(image description generated on this Mac)` … `(end of image description)` — caption section (END marker added)
- `(deep scan — detailed AI description)` … `(end of deep scan)` — deep-scan section
- `(objects detected in image)` — regions section
- `(image verification: …)` — single forensics line

Every writer replaces only its own span. Surya's pass tail re-kicks the caption filler. `caption_model` is cleared only in `update_file_content` (pixel bytes changed), **not** in `set_file_extracted_text` (derived-text upsert) — captions describe pixels, not text.

**D2. One v4 migration, owned by platform, using the feature-spec names.** Keep: `files.caption_model TEXT` (NULL = needs caption), `files.ocr_engine TEXT` + the one-time `user_version<4` backstamp (`ocr_engine='apple'` where text has the OCR prefix — Surya's query-time prefix heuristic is deleted), `image_forensics` (typed BLOB columns for the PNG maps), `image_regions`. Delete: `image_artifacts` (generic JSON table — cannot hold PNG BLOBs, zero real consumers), `image_vectors` + SigLIP asset row (gated OFF by the caption eval; additive DDL later is cheap, premature DDL is forever), `files.captioned_with` (duplicate name). Migration test: legacy room and fresh room converge to the identical `sqlite_master` set.

**D3. The pixel fence gets one choke point (privacy blocker #1) — *rev 2: simpler*.** Today the ":cloud never gets pixels" rule is enforced per-call-site and two sites are open (`locate_in_image`, the post-answer grounding screen check only `is_external_engine`). Because Rust now makes **zero direct Ollama calls**, the fence needs exactly one primary choke point: **sidecar `llm.py`** refuses any generate/vision request carrying `images` when the model is cloud (`:cloud`) or external, raising an `LlmError` that names a local vision model — every present and future image endpoint (`/run`, `/generate`, `/vision_locate`, `/vision_caption`, …) inherits it automatically. Defense-in-depth: the same ~5-line guard in `ollama.rs`'s `sidecar_post` body builders (the single funnel all Rust model calls pass through), plus cloud screens in `locate_in_image`. One regression test per choke point.

**D4. Loopback gate for automatic passes (privacy blocker #2).** `resolved_base_url()` honors the per-room `remote_ollama_url` override — as specced, auto-caption and Surya auto-upgrade would silently upload every imported photo to a remote host while the consent copy says "pixels never leave this Mac." Fix: `ollama::base_url_is_local()`; all **automatic** (non-user-invoked) image passes return early when the resolved URL is not loopback, with a one-line Settings hint. User-invoked actions (Find, Deep scan, attachments) keep today's behavior — the remote override is explicit consent there.

**D5. One download registry, one session cache, one sentinel family.** Three workstreams each cloned `stt_cmds.rs`. Fix: `img_assets.rs` is the single non-Ollama weights channel (streaming `.part`+rename, sha256 verify, one `img-asset-progress` event, one AtomicBool, sentinel `IMG_MODEL_MISSING:<asset>`); forensics and regions register `PACK_SPECS`-style entries and delete their private downloaders. Sidecar-side, `helpers_runtime.py` is the only ONNX session cache (single slot — "one helper resident" actually enforced); `forensics.py`/`florence2.py` expose `load(path)` factories it calls. Platform owns the one `pyproject.toml`/`build-sidecar.sh` onnxruntime addition (lazy imports; 30 s sidecar boot budget untouched). While in there: add the missing sha256 to the Whisper download (15 min).

**D6. License-gate mechanics, unified.** Gated weights (Surya OpenRAIL-M, TruFor non-commercial): explicit "I accept the license" **checkbox** + persisted marker file, then Download. Permissive weights (Florence-2 MIT, SAM 2 Apache-2.0): notice line + plain Download button. Auto-upgrade defaults follow the STT precedent: install-is-consent (ON after an explicit gated download), with a toggle to disable without deleting weights.

**D7. One Settings home.** `ImageIntelligenceSection` (`set-image`, after "AI helpers") is the single home for all optional image weights, license gates, and per-room consent toggles; the existing vision-helper offer moves there; HelpersSection keeps embed/STT. One setting key per behavior (`caption_images`, `img_hebrew_ocr`, `img_zoom`; the dead `img_tamper_on_import` toggle is deleted — forensics is on-demand-only in v1). Existing-room caption backfill is **opt-in**: a one-time strip "Index N existing images in the background? (~H hours) Start / Not now" — never an uninvited hours-long GPU drain on unlock.

**D8. One scheduling rule instead of a scheduler.** The specced `vision_gate` Semaphore, `VISION_TX` third lane, `helper_policy()` ledger, and `/helpers/unload` route had **zero consumers** among the feature designs and are cut. The written rule that replaces them:
- Interactive passes (Find, mark, chat) never wait on background serialization; background yields to interactive — concretely, every background vision loop skips to its yield-sleep while `state.cancels` is non-empty **or** the last ask finished < 60 s ago (`last_ask_at: Instant` in AppState), and defers while `job_cancels` is non-empty.
- Light per-file background vision work = the single-flight caption filler. Anything multi-minute (Surya batches, deep scans) = an ADD-30 job on `Lane::LocalLlm` — job cards, Stop/Resume, checkpoints for free.
- The import OCR lane (`OCR_TX`) carries **Apple Vision only** — Surya never rides it (critics showed a 50-page Hebrew PDF would otherwise block all new imports for ~50 min).
- Fillers chain instead of thrash: caption filler drains its batch fully, then kicks the summary filler **once**, and defers while the embed backfill has a large pending batch.
- TruFor (up to ~4 GB working set) is mutually exclusive with Ollama vision passes on < 32 GB: verify sets a static flag the vision loops check, and refuses to start while a grounding/deep-scan call is in flight; its `max_pixels` defaults to 1.0–1.5 MP on 16 GB (2 MP at 32 GB+).

**D9. One image-fitting implementation, in Python — *rev 2: consolidated*.** The original design implemented aspect-preserving fitting twice (`vision.rs` + `vision.py`) with a cross-language parity test — necessary only because both stacks talked to Ollama. They no longer do. Per the migration doctrine ("all model I/O and prompt logic live in Python; Rust gathers bytes and stores results"), **`vision.py` owns the single `fit_image`** with two profiles: `GROUNDING_MAX_PIXELS = 2_097_152` (Ollama Go-engine GGUF default — verified there is **no** request/env/Modelfile knob; we cap client-side at the cross-version minimum), `GROUNDING_MIN_PIXELS = 802_816` (llama.cpp: Qwen-VL grounding needs ≥1024 image tokens), `ATTACH_MAX_PIXELS = 1_003_520`; EXIF orientation via `ImageOps.exif_transpose`; byte-passthrough for in-range PNG/JPEGs. **Rust deletes `prepare_image`/`VISION_SQUARE` entirely** and sends original bytes everywhere; the sidecar fits user-attached images on `/run`/`/generate` ingress to the ATTACH profile and fits grounding inputs inside `/vision_locate`. The parity test disappears with the duplication. Rust still *renders* zoom crops and PDF pages (that is DB/extraction work, not model I/O) but never resizes for model consumption. DC2 sets its tile budget to `GROUNDING_MAX_PIXELS` (halves its model calls vs its private 1.05 MP constant); zoom documents its 1344 px crop cap as 48×28 within the attach budget — both import the Python constants.

**D10. Event and bench conventions.** All new progress events carry `{id, name, stage, …}` keyed by **file id** (the name-keyed variant re-introduces a known rename-orphan bug). One bench tree: `sidecar/bench/img_bench.py` suites (vstar, docvqa, hebrew_ocr, capret, tamper — added as their features ship) + one `fetch_subsets.py` + one RESULTS template that every phase fills before merging; the seven per-design harnesses become suites, not parallel skeletons.

**D11. The qwen2.5vl tag/license decision (decide in P0, one owner).** The bare `qwen2.5vl` tag pulls the **7B (~6 GB, Apache-2.0)** while the offer copy says "~3 GB"; the 3B is under the **Qwen Research License** (non-commercial), not Apache (one design's claim to the contrary is corrected). Recommendation for a 16 GB target: pin `recommended_models.vision = "qwen2.5vl:3b"` and add a one-line Research-License notice to every offer surface. The offer button is the license-acceptance surface — this is consent accuracy, not pedantry.

**D12. Surya never leaks into vision surfaces, and its pull is integrity-checked.** Surya's GGUF answers `/api/show` with the "vision" capability, so without exclusions it would be selectable as a describe model (OCR-HTML gibberish in chat) and would satisfy "can see images" UI checks. `is_ocr_helper_model()` is honored in `can_read_images()`, the `perceive_image` fallback loop, `best_default`/`best_local_default`, and every frontend `caps.some(vision)` derivation. Supply chain: `ollama pull hf.co/...` cannot pin a revision, so after pull the local manifest digest is verified against a compiled-in expected digest (mismatch → delete + error); if digest reading proves brittle, the direct-download + `ollama create` path (Plan B) becomes primary.

## Part 2 — The workstreams

Each chapter summarizes the full function-level spec (preserved verbatim in the workflow transcripts) with the review amendments applied.

### W1. Native resolution + capability truth (quick wins) — 3.5 days — Phase 0

**What ships (rev 2 — Python-owned fitting, Rust deletions).** (1) The 1000×1000 stretch dies **once, in `vision.py`** (D9): an aspect-preserving `fit_image` with GROUNDING (min 802,816 px, max 2,097,152 px) and ATTACH (max 1,003,520 px, token-neutral vs today) profiles — byte-passthrough for in-range PNG/JPEGs (zero resample), /28-rounded downscale mirroring Ollama's SmartResize (so the server resample is identity), and **EXIF orientation applied** via `ImageOps.exif_transpose` (fixes a live box-misalignment bug on iPhone photos). Small images upscale to the grounding minimum — llama.cpp documents Qwen-VL grounding degrading below 1024 image tokens. Rust-side: **delete `vision.rs` `prepare_image`/`VISION_SQUARE`**; `gather_context_and_save_question` (agent.rs:414) and the `mark_image` tool arm (:1754) send original bytes; the sidecar fits on ingress. (2) The post-answer grounding pass (agent.rs:265–306) stops hand-rolling `grounding_prompt` + `chat_structured` + `parse_boxes` in Rust — per migration doctrine it **calls `/vision_locate`** (which already owns prep + prompt + schema + parse in Python) and stores the returned boxes; the Rust grounding helpers become dead code and are removed. This also makes W4's crop-grounding branch a one-argument change. (3) One capability taxonomy in `models.rs` (model *management* stays Rust per the migration spec): `is_grounding_model()` (curated Qwen-VL allowlist), async `can_read_images()` (authoritative caps via the existing sidecar `/capabilities` passthrough, process-cached; name-heuristic fallback only when caps are unknown; **false for cloud/external/OCR-helper models** — feeds the D3 fence), `ModelCaps.grounding`. (4) Frontend gates key on *grounding*, not *vision*: installing gemma3 no longer hides the qwen2.5vl offer; degraded grounding is labeled "approximate" in the viewer, tool results, and ask-steps.

**Key facts pinned by web-check:** Ollama's qwen25vl engine does aspect-preserving SmartResize to /28 with `vision.max_pixels` default 2<<20 — set in GGUF metadata, not settable per-request; hence the client-side cap. Coordinates were never skewed by the stretch (normalized round-trip cancels) — the harm was detail loss and double resampling; the frontend overlay contract (normalized 0..1) is untouched.

**Perf:** grounding prefill worst-case ~2× on over-cap images (est. +1.5–4 s on base M — the price of actually reading retina screenshots); chat attachments token-neutral; photos ≤2 MP get **faster** (byte passthrough). One consideration the consolidation adds: original bytes (up to the 50 MB viewer cap) now cross the loopback IPC instead of a ~1–3 MB canvas PNG — bounded, loopback-only, acceptable; the sidecar thumbnails before Ollama sees anything. **Tests:** sidecar `fit_image` suite (passthrough, /28 rounding, profile min/max, EXIF-6 swap, BMP transcode), capability matrices incl. `:cloud` refusal, `/vision_locate`-routed grounding regression, and the `eval_grounding` IoU harness (8 synthetic cases incl. EXIF-rotated and 3000×2000) — acceptance: median IoU ≥ old pipeline everywhere, strictly better on the over-cap and EXIF cases. **Amendments:** owns the D3 choke-point fence and the D11 tag decision; implements D9 (single fitting, Rust deletions); its `first_image` restructure and `/vision_locate`-routed post-answer block land first per the Part 3 merge order. Effort drops 4 → 3.5 days (one implementation, no parity machinery, net-negative Rust diff).

### W2. Caption-at-import + semantic image search — 6 days — Phase 1

**What ships.** Every image gets a short structured description — English caption + Hebrew caption + up to 12 object tags — generated on-device by `vision_model()` (qwen2.5vl, or the multimodal default chat model). The caption block is upserted into `extracted_text` via D1's section helper, so **FTS5, nomic embeddings, retrieval blending, chat context, and the one-liner summary all work with zero retrieval-side changes** — "find the photo with the whiteboard" just works. A background "caption filler" cloned from the proven `spawn_summary_filler` skeleton (single-flight, 45 s delay, D8 yield rules, short keep_alive) drains `files.caption_model IS NULL` at ≤200/run. New imports caption automatically (OCR precedent); **existing rooms get the opt-in strip** (D7) with an honest count + time estimate. UI: a pulsing "describing…" FileRow badge keyed by file id; a Privacy/ImageIntelligence toggle (`caption_images`, default ON) + "Describe images now" button; the loopback gate (D4) silences it when a remote Ollama is configured.

**Track (b) — CLIP/SigLIP image vectors — is deliberately NOT shipped in v1.** It would add a second embedding space, an onnxruntime dependency, a 0.4–1.5 GB download, and per-question latency for unproven recall gains (the research had no verified claims here). Instead the `capret` eval suite ships with a **pre-registered gate**: if caption-only recall@5 ≥ 0.8 on object/scene queries and ≥ 0.6 on fine-attribute queries, captions suffice; below that, the fully-written deferred spec activates (SigLIP2-base, Apache-2.0, multilingual text tower — chosen over MobileCLIP2 (murky apple-amlr license) and jina-clip-v2 (CC-BY-NC); int8 ONNX ~400 MB via `img_assets.rs`; separate `image_vectors` table; RRF third signal; +7–9 days).

**Perf:** zero resident additions; 5–16 s/image on base M (platform's honest numbers — the 2–6 s claim is corrected); a 50-image import ≈ 5–13 min of background work. **Degradation:** every failure path leaves `caption_model` NULL and retries on the next trigger; background paths are silent, the explicit button surfaces sentinels. **Tests:** FTS round-trip incl. nikud-stripped Hebrew, idempotent re-caption, migration test, sidecar endpoint tests, and the real-photo eval room (never synthetic-only — the rec_bench lesson). **Amendments:** uses D1 markers (END marker added), D9 ATTACH profile (its private `caption_prepare_image` is deleted), D4 loopback gate, D7 opt-in backfill, D8 filler chaining; a vision-model pull completion now kicks `caption_images_now` (was a spec gap).

### W3. Deep scan (DC2 divide-conquer-combine) — 6 days — Phase 2, gated

**What ships.** A user-invoked "Deep scan" button in ImageView (plus a passive hint chip on ≥4 MP images): a durable ADD-30 background job that captions the whole image at the model view, partitions the native-resolution image into 2–12 overlapping tiles each fitted to the D9 budget, captions each tile with the **same** resident model (global caption as context, "NOTHING NOTABLE" escape hatch, verbatim text transcription incl. Hebrew RTL), and fuses the notes into one detailed description upserted as the deep-scan section — indexed for retrieval automatically. A 12 MP photo goes from being read at ~8 % of native detail to ~25–50 %. Auto-scan-at-import is **rejected** (a 20-photo import would hold the LocalLlm lane for an hour). Progress/Stop/Resume ride the existing job cards; fusion failure falls back to storing the region-tagged concatenation.

**Perf:** 4 MP ≈ 1.5–3 min; 12 MP ≈ 3–5.5 min (halved from the original spec by adopting the D9 2.1 MP tile budget); model resident only for the job's duration via `vision_keep_alive`. **Gate (per review):** ships after zoom, and only if `img_bench` shows retrieval misses on high-res fixtures that captions+zoom don't cover — zoom already handles the interactive "read the small detail" case with the same model and no new weights. The full spec is ready either way; its `/deep_scan_caption` endpoint merges with `/vision_caption` (one endpoint, optional `region_tag`/`global_caption` params). **Amendments:** D9 constants (private `MODEL_VIEW_PX` deleted), D1 section helper, D8 job-lane rule, grounding-after-zoom double-upscale fix (skip the min-pixels upscale when input is already a rendered crop).

### W4. Agentic zoom ("look closer") — 8 days — Phase 2

**What ships.** The superhuman headline: a `look_closer` agent tool. When a question concerns a small/distant detail, the model iteratively proposes a region of its current view; Rust's `exec_tool` (tools execute Rust-side via the MCP bridge — unchanged by the migration) composes viewports and crops the **original bytes at native resolution**; the crop rides the existing, tested `pending_images` → `room_mcp` image-block → LangGraph user-turn pipeline, and the sidecar fits it to the model budget (D9) — the already-warm multimodal chat model perceives its own zooms with **zero extra model residency**. Search policy is pragmatic self-report (Ollama exposes no logprobs — no Zoom Eye tree scoring) with deterministic guardrails: 6 calls/turn, 12 % context padding, min-crop floor (~×40 max linear zoom), `reset:true` backtrack, stale-crop retirement keeping ≤2 images in context. When a zoom trail exists, the post-answer grounding pass runs `/vision_locate` on the **final crop** (a one-argument change on W1's rerouted block) and maps boxes back to full-image coordinates — needle objects become markable that full-image grounding misses. UI: live "Looking closer (×3.2)" step chips; the zoom trail persists in `messages.effects` and renders as numbered dashed outlines in the chat bubble.

**Rev 2 — native-loop half deleted, dual maintenance gone.** The original spec targeted both engines: a native `agent_loop` injection path *and* the sidecar routing. The native loop no longer exists, so the tool is delivered **only** the way every agent tool now is: `room_mcp` `served_tools` pushes the spec under `scope.include_app_tools()`, `routing.py` gains `ZOOM_TOOL_NAMES` + hints (parity-tested against the Rust hint source), and `graph.py`'s prepare filters it unless the routing flag fires. The stale-crop retirement policy — previously duplicated in Rust and Python, flagged as a dual-maintenance risk — now lives **only in `graph.py`**. The Rust remainder is exactly the `exec_tool` arm + the pure `vision_zoom.rs` geometry module (session, viewport composition, crop rendering, trail effects). Effort drops 9 → 8 days.

**Privacy:** the tool rides the served-tools scope fence (never `tools_catalog`), so cloud-advisor scopes structurally cannot see it; `room_mcp` emits image blocks only for the LocalEngine scope, and the D3 sidecar choke point refuses pixels to `:cloud` models regardless. **Ship gate:** `zoom_bench` (a suite in the D10 bench tree) — ~25 V*Bench needle items (dev-fetched only; license unspecified, never redistributed) + ~15 self-shot committed Hebrew RTL fixtures + 10 controls; zoom-enabled must beat single-pass by **≥ +15 pp on needles with zero control regression**, median added latency ≤ 20 s. **Amendments:** rebases on W1 (`can_read_images`, `first_image` shape, Part 3 merge order); trims its over-broad hint list (generic tokens like "say"/"small" would fire on ordinary questions and freeze into the parity-tested sets); fixture fetching merges into the shared bench tree.

### W5. Hebrew OCR upgrade (Surya 2) — 7 days + 1 contingency — Phase 3

**What ships.** A download-gated, license-noticed OCR upgrade riding entirely on existing Ollama plumbing: `ollama pull hf.co/datalab-to/surya-ocr-2-gguf` (surya-2.gguf 1.27 GB + mmproj 205 MB, verified on the HF repo) through the existing pull-progress UI, inference through the existing sidecar `/generate` gateway with the model's exact training-contract prompt ("OCR this image to HTML…" — verbatim from `prompts.py`, not paraphrased), HTML→text post-processing (drop Figure/Blank-Page divs, tables → ` | ` rows, entity decode). Import flow: Apple Vision text is written **first and always survives**; when Surya is installed and the text contains Hebrew (U+0590–05FF) or came back empty, a background Surya pass re-reads page-by-page and replaces **only the OCR section** (D1). Provenance in `files.ocr_engine`. Existing rooms: "Re-read N existing scans" — explicit button, never automatic.

**Runtime reality:** ~1.9 GB resident while OCRing (helper-class, `KEEP_ALIVE_SHORT`, never pickable as a chat model per D12); ~1 min/page budget (datalab's Apple Silicon p50 is 59 s at parallel-8 — the Phase-0 spike re-measures single-stream on 16 GB). **License:** weights are modified OpenRAIL-M (free personal/research/<$5 M revenue + non-compete) — D6 checkbox gate with honest copy; the app never redistributes weights. **Amendments (structural, from the blocker):** every Surya pass — import auto-upgrade *and* bulk re-read — moves off the OCR lane onto **ADD-30 `Lane::LocalLlm` jobs** (one page per step, checkpoint/resume; the bespoke cancel AtomicBool, custom progress event, and Settings progress strip are deleted — job cards provide all three); D12 digest verification after pull; D12 capability exclusions everywhere; legacy-candidate detection keys on the D2 `ocr_engine` backstamp (prefix heuristic deleted). **Spike (0.5 d, blocking):** confirm the hf.co pull fetches the mmproj and Ollama's engine runs qwen3.5-arch multimodal GGUFs; Plan B (direct download + `ollama create`) is fully specced; Plan C (llama-server helper process) documented only.

### W6. Image forensics (TruFor) — 13 days — Phase 4, legal-gated

**What ships.** A "Verify" button in ImageView that detects what eyes cannot see: TruFor (CVPR 2023; independently top-ranked in IMDL-BenCo) runs fully on-device via a dev-side torch→ONNX export (torch never ships; onnxruntime + numpy add ~80 MB to the sidecar bundle, lazy-imported). Three honest outputs: a per-pixel tamper heatmap canvas overlay (rides the existing zoom), a whole-image integrity score presented as **three calibrated bands — never "FAKE"** ("editing traces detected" / "inconclusive" / "no traces found — note this does not prove the image is unedited"), and the reliability map rendered as default-ON dimming of low-confidence regions. Results cache in `image_forensics` (sha256-keyed, cascade-delete); a one-line provenance summary joins `extracted_text` so "which images show editing traces?" works in chat. Works with Ollama entirely absent. Videos are refused with a clear message (keyframe sampling is a named Phase-2 follow-up, as is a `verify_image` agent tool).

**The two hard gates, front-loaded:** (1) **Legal, at day 0** — the weights are "informational and nonprofit purposes only" (GRIP-UNINA), and self-hosting a converted ONNX artifact is redistribution of a modified work; the named fallbacks are *not* cleaner (Mesorch's checkpoints embed NVIDIA non-commercial SegFormer weights). No export engineering starts until this is answered. (2) **RAM (blocker fix)** — `max_pixels` defaults to 1.0–1.5 MP on <32 GB (2 MP at 32 GB+), and verify is mutually exclusive with background vision passes per D8; the corrected budget includes the ~5.5 GB OS/app floor. **Eval ship-gates:** AUC ≥ 0.85 on a ~120-image local set (incl. Hebrew document screenshots — a known false-positive class), pristine-flagged-suspect ≤ 10 %, CoreML-vs-CPU map divergence < 2e-2 (else CPU default). EXIF-transpose before inference is load-bearing (the map must align with the displayed pixels). **Amendments:** downloads via `img_assets.rs` (private downloader deleted); session cache via `helpers_runtime.py`; D6 checkbox gate; D2 typed-BLOB table confirmed over the JSON alternative.

### W7. Pixel-perfect localization (Florence-2 + SAM 2) — 16.5 days — Phase 5, evidence-gated

**What ships (fully specced, as requested — with the review's gate honored).** "Outline every signature on this page" as exact polygons: (Stage 1) **Florence-2-base-ft ONNX int8** (~275 MB, MIT) in the sidecar — `OPEN_VOCABULARY_DETECTION` finds every instance of a free-text phrase; `REGION_TO_SEGMENTATION` emits polygons directly as location tokens (no mask tracing); ImageView's Find bar upgrades to polygon SVG overlays; a `find_regions` agent tool draws them in chat via `effects.regions`; import-time `DENSE_REGION_CAPTION` labels ("signature; stamp; logo") join `extracted_text` so "images containing signatures" is searchable. (Stage 2) **SAM 2.1 hiera-small ONNX** (143 MB, Apache-2.0 — code *and* checkpoints, the BSD-3 claim was corrected): box-prompted pixel-exact masks → simplified polygons, offered only after Florence.
