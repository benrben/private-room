# Master Implementation Plan — 2026-07-18

Step-level implementation plans for all 13 backlog ideas, derived from the verified
[ideas triage](ideas-triage-2026-07-18.md). Each wave was planned by a dedicated agent
that read the cited code, then adversarially reviewed by an independent critic that
re-read the code hunting for wrong anchors, missed wiring, invariant violations, and
schedule risk. Confirmed findings appear as **Review amendments** after each plan —
executors must apply the amendments as part of the plan; they are not optional notes.

Scope upgrades baked in (2026-07-18): Idea 1 = full external-agent parity
(Claude Code / Codex / Flowed get everything the in-room agent can do, incl. file_pass);
Idea 2 = accepted flagship (dedicated on-screen Workflows page, saveable custom-workflow
library, animated visual pipeline, agent-authoring tools).

**Total ≈ 62–73 person-days** (13 ideas) of focused solo work (agent-accelerated execution changes
wall-clock, not the review/verify burden). Waves are independent except: 7 depends on 4;
6 builds on 4's plumbing; the workflows queue (2/M1) supersedes the one-job guard that
idea 8's auto-jobs and idea 1's external `start_file_pass` both bump into — landing
2/M1's queue early would remove those collisions, but it is NOT a hard dependency.

## Before any wave: land the current tree

`moonshot-impl` carries a large uncommitted working set (the 31-fix security wave, the
sidecar consolidation, TitaNet work). Commit it (or split it into reviewable commits)
before starting new feature branches, or every wave's diff will drag that baggage.
Build/verify recipe per wave (from project memory): sidecar rebuilds need
`build-sidecar.sh --clean`; `tauri build` exit-1 at the end is the missing updater key
(fine locally); run `scripts/macsign.sh` after every build or TCC grants (mic, etc.)
reset; `xattr` shim gotcha → build with `PATH=/usr/bin:$PATH`.

## Verification cadence

Every wave ends with: `cargo test` (workspace) + sidecar `pytest` green; the plan's
per-idea end-to-end verification steps performed in the built, signed app; and for
UI-bearing waves a computer-use QA pass over the new surfaces (the embodiment-loop QA
harness from v0.2.3 applies directly).

## Idea 7 — Multi-File Atomic Operations (`edit_files`)

**Goal** One batched `edit_files` tool that validates every edit up front and applies all of them in a single `BEGIN IMMEDIATE` transaction, so a five-file refactor either fully lands or fully doesn't — with all batch snapshots sharing a cause tag for group visibility/undo. Follows `set_cells`' validate-everything-then-write shape (agent.rs:1464-1515) and `restore_file_version`'s transaction discipline (safety.rs:32-44).

**Depends on** Idea 4 (uses `compute_edit` and its uniqueness-guarded matcher; shipping batches on top of the old exact-only matcher would multiply the failure rate by N).

**Steps**

1. **(M)** Batch executor in `src-tauri/src/commands/edit_match.rs`:
   ```rust
   pub(crate) struct BatchEdit { pub name: String, pub old_text: String, pub new_text: String }
   pub(crate) fn run_edit_files(conn: &Connection, edits: &[BatchEdit])
       -> Result<BatchApplied, String> // { batch_id, files: Vec<(file_id, real_name, count)> }
   ```
   - Cap: `const MAX_BATCH_EDITS: usize = 20;` reject larger with guidance.
   - **Phase A — validate, zero writes:** iterate edits in order keeping `working: HashMap<file_id, (real_name, Vec<u8>)>`; for each edit resolve via `db::find_file_like`, load bytes from `working` or `db::get_file_bytes`, run Idea 4's `compute_edit` **against the working bytes** (so two edits to the same file chain, exactly like `set_cells` chains `set_cell_in_bytes` over `bytes` at agent.rs:1501-1505), store the result back in `working`. Any failure → `Err(format!("Edit {i} of {n} ({name}): {inner}"))` — nothing written, and the inner message keeps the `closest_snippet`/ambiguity hints so the model can fix just that edit.
   - **Phase B — write:** `batch_id` = first 8 chars of a new UUID; `conn.execute_batch("BEGIN IMMEDIATE")`; for each entry in `working` (one per **file**, final bytes): re-derive text via `extraction::extract_text`, then `store_file_bytes(conn, id, bytes, text, &format!("AI edit (batch {batch_id})"))`; `COMMIT`, on any error `ROLLBACK` and return the error — byte-for-byte the match/rollback shape of safety.rs:38-44. # Wave 4a — Workflows Flagship (Idea 2)

## Idea 2 — LLM Graph Workflows + Scheduler + Flagship Workflows Page

**Goal**: Give Private Room a first-class automation surface: users (and the in-room agent) compose multi-step LLM pipelines from a curated node palette, save them as named reusable workflows, run them manually or on a schedule, and watch them execute as an animated visual pipeline on a dedicated Workflows page. All orchestration runs in Rust on the existing ADD-30 jobs engine (persisted, checkpointed, resumable); the sidecar's stateless HTTP endpoints (`/generate`, `/summarize_file`, `/file_pass_map`, `/run`) are the node primitives — no dynamic LangGraph composition.

**Depends on**: none hard. Touches the same guard sites as Idea 1 (the one-job guard becomes a queue, which also erases Idea 1's collision risk) and Idea 8 (the new-file-summarizer template supersedes its auto-trigger). Land this before or independently of both.

# Wave 4b — Supernatural Voice MVP (Idea 3)

## Idea 3 — Custom Supernatural Voice (MVP: AVSpeechSynthesizer + Web Audio DSP archetypes)

**Goal** Give the room's AI an audible voice: every assistant reply can be spoken aloud through a user-picked "supernatural archetype" (demon / ghost / ancient entity) implemented as synthesis-side pitch+rate settings plus a Web Audio DSP chain at playback, fully on-device with zero model downloads. Audio starts before the full reply lands via sentence-chunked synthesis fed from the existing `ask-delta` stream, with per-message play buttons and a per-room auto-speak toggle. The neural-TTS tier (Piper/Kokoro/ONNX voices) is **explicitly out of scope** — it is a future decision to take only after this DSP MVP proves the feature (triage: Piper's espeak-ng is GPLv3; tract-onnx is validated for TitaNet only, per the warning in `src-tauri/Cargo.toml:58-62`).

**Depends on** none (touches no other idea; engine-agnostic because it taps the `ask-delta` event stream, not the sidecar).

### Synthesis-approach decision (investigated, binding)

**Pick: AVSpeechSynthesizer's write-to-buffer API via `objc2-avf-audio`, not `NSTask say -o`.** Verified feasible: crates.io has `objc2-avf-audio` 0.3.2 whose `AVSpeechSynthesizer` exposes `writeUtterance_toBufferCallback` behind the `AVSpeechSynthesis` feature (checked against docs.rs). Justification over `say -o`:

1. **Pitch control exists only in the API.** `AVSpeechUtterance.pitchMultiplier` (0.5–2.0) is the base layer of every archetype (demon pitches down at synthesis, keeping duration correct); `say` has no pitch flag, which would force all pitch work onto `playbackRate` (pitch and tempo coupled).
2. **No decrypted text/audio on disk.** `say -o` must write a temp file — spoken assistant text is room content, and the app's doctrine is to sweep such artifacts (`commands::cleanup_browser_previews` in `lib.rs:25`). Write-to-buffer keeps PCM in memory end to end.
3. **Latency.** Sentence-chunked synthesis spawns per sentence; a subprocess per sentence adds ~100–300 ms each, in-process synthesis doesn't.
4. **The codebase already owns this exact pattern.** `snapshot.rs:23,72` (block2 `RcBlock` completion handler + `#[cfg(target_os = "macos")]` with a non-mac stub at `snapshot.rs:131`) and `ocr.rs:61-70` establish the objc2 style; `stt_cmds.rs:144` (`transcribe_audio`) establishes base64-audio-over-IPC + `spawn_blocking`.

Known gotchas to code for: buffers arrive as `AVAudioBuffer` on an internal queue → downcast to `AVAudioPCMBuffer`, handle both Float32 and Int16 `commonFormat`; a zero-`frameLength` buffer signals end-of-utterance; keep the `AVSpeechSynthesizer` retained until done and guard with a 30 s watchdog on the mpsc receive. AVSpeechSynthesizer needs no TCC permission or entitlement (unlike mic — `private-room-mic-hardened-runtime` doesn't apply).

# Wave 5 — Runnable & Schedulable Scripts (Idea 13, added 2026-07-18)

## Idea 13 — Scripts in the room: one-click run, nightly schedule, header shortcuts

**Goal**: A `.py`/`.js` file stored in the room is a first-class *script*: runnable with one click from the file viewer header or a dedicated Scripts page, schedulable through the Wave 4a workflow scheduler (e.g. `update_portfolio.py` pulls quotes via yfinance every night and writes the CSV back into the room), with declared dependencies/inputs/outputs surfaced in a consent card before anything executes. Because a child process can never read the SQLCipher room DB, every run materializes its declared inputs into a throwaway workspace, runs there, and imports outputs back through `store_file_bytes` — so script writes are versioned and undoable like every other edit.

**Depends on**: Wave 4a **M1** (workflow engine: `WorkflowDef`/`NodeKind`, `compile_workflow`, `execute_workflow_step`, `queue::submit`/`pump`/`start_job_from_row`, `db/workflows.rs`, `run_workflow`) is a hard dependency — the runner is a new node kind, not a parallel job system. The schedule half additionally needs **M4** (`scheduler.rs` tick + catch-up) and reuses M2's `SchedulePopover` (plan step 9). All accepted 4a amendments apply (single-node script workflows are trivially dense, so the done-set [BLOCKER] costs nothing here; no child jobs are created, so `parent_job_id` is untouched). Shared-component dependency: the header **ShortcutBar container is shared with Idea 2's header shortcuts** — whichever lands first names the component (`src/workspace/ShortcutBar.tsx`, props `{ items: {id,label,icon,title,run}[] }`); this plan only defines the Scripts-side items feeding it.

### Binding design decisions (address the hard constraints up front)

1. **I/O bridge — materialize → run → import-back.** Room files live inside the encrypted DB (`db/schema.rs:13-26`, SQLCipher key applied in `apply_key`, schema.rs:142-145); a spawned interpreter cannot see them. Per run: create `app_cache_dir()/script-runs/<job_id>/` with mode `0o700` (the `0o600` discovery-file precedent from Wave 1a step 5 extends to dirs), write each declared input's bytes under its real room name (`db::get_file_bytes`, db/files.rs:189; resolved via `db::find_file_like`, db/files.rs:338 — newest match wins, same as the agent's tools), run the script with `cwd` = that dir, then import back (decision 2) and **delete the dir in the epilogue on every outcome** (a startup sweep in `lib.rs` setup also removes orphaned `script-runs/*` from crashes, the `quiesce_stale_jobs` spirit, jobs.rs:208-218). Consequence to state honestly (in plan and consent card): inputs sit **unencrypted on disk for the run's duration** — same trust class as Export (ViewerPane.tsx:191-198, which is `data-agent-blocked` for the same reason). Room mutations happen **only in the import-back phase after exit 0**, so Stop/kill/timeout/crash never leaves partial room writes — the run is transactional from the room's point of view.
2. **Import-back rules + auto-import (decided: yes, capped).** A declared output whose name matches an existing room file → `store_file_bytes(conn, id, bytes, extract_text(...), "Script ran — <script name>")` (files.rs:323-332) — snapshot-then-overwrite, so **every script run is undoable via Time Machine for free**. A declared output that doesn't exist yet, **and any NEW file the script created in its workspace** (present after exit, not among the materialized input paths) → `db::insert_file(..., source="script")` (db/files.rs:4-21; `source` is free-text, no migration). Auto-import is required, not optional: the flagship example writes computed names (`prices-2026-07-18.csv`) that literal declarations can't cover, and new files are *additive* — worst case is clutter, capped at **20 files / 64 MB per run** (beyond the cap: skipped + named in the run log). By contrast, **overwriting an existing room file stays declaration-gated**: an input modified in place but absent from `room-outputs` is *not* written back (logged in the artifact) — destructive writes must appear on the consent card. `room-files-changed` is emitted after import (the publish-arm precedent, file_pass.rs:434-436).
3. **Manifest syntax (exact).** Parsed from the first 64 lines; comment prefix `#` for `.py`, `//` for `.js`; first occurrence of each key wins; keys case-insensitive:
   ```python
   # /// script
   # dependencies = ["yfinance", "pandas"]
   # ///
   # room-inputs: portfolio.csv, holdings.xlsx
   # room-outputs: portfolio.csv
   # room-timeout: 300
   # room-shortcut: global
   ```
   `room-inputs`/`room-outputs`: comma-separated literal room file names. `room-timeout`: seconds, clamped 5–3600 (default 600 — first `uv` runs resolve and download wheels). `room-shortcut`: `global` (TopBar) | `file` (default when inputs/outputs are non-empty) | `none`. Python dependencies are **PEP 723 inline metadata** — chosen because `uv` executes it natively, so we never write a requirements format of our own; our parser reads the `dependencies = [...]` line only tolerantly, for display and for the has-deps decision — `uv` is the authoritative parser at run time. JS: `// room-…` forms; no dependency key in v1 (decision 4).
4. **Runtime resolution (v1 + fallback).** `.py`: prefer **`uv`** when installed — resolved by the `markitdown_extract` candidate-list pattern (extraction.rs:90-110) + the `ollama_bin` login-shell fallback (`zsh -ilc 'command -v uv'`, ollama_lifecycle.rs:109-133); invocation `uv run --no-project <script.py>` (reads PEP 723, per-script cached env, no project pollution). Without `uv`: `/usr/bin/python3` (then `python3` candidates) **only for scripts with an empty dependency list**; deps + no uv → the actionable error "This script needs yfinance, pandas. Install uv (`brew install uv`) to run scripts with dependencies." (ADD-22 corrective-error doctrine). `.js`: run only if `node` resolves (same candidate pattern: `/opt/homebrew/bin/node`, `/usr/local/bin/node`, login-shell) and the script is dependency-free; otherwise the same actionable error. App is hardened-runtime but **not** sandboxed — spawning is exactly what `ollama_lifecycle::ensure_up` (:166-172) and `sidecar_lifecycle::spawn_and_wait` (:118-123) already do. Open decision 1 covers per-room persistent venvs.
5. **Consent (SEC-1 doctrine, exactly).** The room's author is the attacker (mcp_cmds.rs:43-45), so script approvals are **per-Mac, content-addressed, and never inside the `.roomai`**: `script_approvals.json` in `app_data_dir()` (clone `mcp_approvals_file`, mcp_cmds.rs:46-72) holding SHA-256 fingerprints of script *bytes* (clone `mcp_fingerprint`, :27-32). Any edit changes the hash → re-prompt, for free. Manual first run: an inline gate cloned from `mcp_call_approved` (:247-290 — emit card, 180 s timeout, timeout = decline) with a card cloned from Overlays.tsx:77-116 (`data-agent-blocked` — the UI-driving agent must never approve its own script). **Scheduled runs are pre-consented at schedule time** and the executor never blocks on a modal: the job plan snapshot carries `consented_sha256` (stamped at enqueue — the just-approved hash for manual runs; from the approvals file at scheduler fire time); the executor re-hashes the script and on mismatch **parks the run as error** "Script changed since it was approved — review it on the Scripts page" + toast action. This implements the Second-Pass Audit's ✅-confirmed approval-gates policy (plan lines 73-75): approvals pre-granted at save/schedule time, or park — never a 180 s hang inside an unattended job.
6. **Lane + kill semantics.** `Lane::Cpu` (jobs.rs:33-40) — the run is subprocess-bound, a workflow with one script node occupies the queue's single `running_job` slot anyway (4a step 4), and a new `Lane` variant would churn every serialized plan for zero scheduling benefit. Spawn with `std::os::unix::process::CommandExt::process_group(0)` so `uv`'s python child dies with it; watcher loop polls `child.try_wait()` every 250 ms against the job's cancel flag (jobs.rs:189-194) and the manifest timeout; on either: `kill -TERM -- -<pgid>` (house `Command::new("kill")` pattern, ollama_lifecycle.rs:213), 5 s grace, `kill -KILL -- -<pgid>`. Cancel → step `Err` → the spawn epilogue's cancel-normalization makes it `Paused` (the spawn_file_pass shape, jobs.rs:801-804); timeout → `Error("script timed out after Ns")` with Retry on the card. Stdout/stderr are drained on two reader threads (the `BufReader`-on-blocking-thread pattern, sidecar_lifecycle.rs:132-143) into 32 KB ring tails.
